# Stage 1 — Orient

*Setup Arc stage reference. Full prose in [playbook.md → Stage 1](../playbook.md#stage-1--orient). The engine reads this file to run the stage.*

## Purpose

Before setting a single goal, answer two questions: *What game are you playing?* and *What do you actually want?* Skip this stage and the user spends the year executing brilliantly against the wrong target. Orient produces the one-sentence Direction that is the input to every stage after it.

## Framework: Finite vs. Infinite Games

Finite games have winners, losers, and a finish line. Infinite games have no endpoint — the goal is to keep playing well. A business is an infinite game: more like marriage, parenting, or gardening than football. When a user sets finite-game goals inside an infinite game, they optimize for short-term wins at the expense of long-term sustainability — burning out the team chasing quarterly targets, damaging customer relationships hitting revenue goals.

Use this frame to test whether the user's stated want is sustainable, or a short-term win that costs the long game.

## Diagnostics: the three tests every goal must pass

Run the user's candidate direction through all three. These are the bar — reference them **by name** when pushing back at any later stage.

1. **The 30-second test.** Can the user explain this to a team member, a spouse, a stranger at a coffee shop in 30 seconds or less? If it needs a PowerPoint, it's too complicated to drive behavior.
2. **The human-aware test.** Does it assume superhuman conditions — constant energy, no sick days, instant process adoption, perfect information, linear progress? If yes to any, it's machine logic on a human system.
3. **The should-want vs. actually-matters test.** Is the user chasing this because it'd make them look successful, or because achieving it would actually move the business forward? Should-want goals sound impressive, produce no joy, and almost never get achieved.

## Difficulty calibration

A goal too easy generates no pull; a goal impossible generates anxiety and paralysis. The bar: **challenging but possible.** If the user would bet money on hitting it without changing anything, it's too easy. If they can't imagine a path to it even with effort, it's too hard.

## The warning: vision boards are not goals

Writing down what you want and putting it on a corkboard *feels* like progress and isn't — the visualization feels like achievement and takes the edge off the drive to do the work. Clarifying what you want is fine; confusing it with achieving it is the trap. If the user's "direction" is really a wish-list, name it.

## Posture notes

- This is where the **friction half** earns its keep. A vague want ("grow the business," "be successful") should not survive this stage unchallenged. Push it through the three tests until it's specific.
- Stay in the **lane half**: *which* game the user wants to play and *what* they want are theirs to decide. You sharpen the statement; you don't pick the destination.

## Deliverable

A one-sentence Direction that survives all three tests. Plain language, no jargon. Write it to `goals/vision.md` under a `## Direction` heading, noting which tests it was checked against. This is the input to Stage 2.
