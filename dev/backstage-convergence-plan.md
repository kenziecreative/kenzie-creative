## Rules for Maintainer Agents

1. **Respect per-plugin design intent.** Strategist deliberately has no evidence layer and advisory-not-blocking gates (its AGENTS.md says so). Goal-setting deliberately has no outputs gate. Do not "fix" deliberate omissions — converge the engagement architecture, not the domain design.
2. **Adapt language, not structure.** Each plugin keeps its own voice (research partner / strategy advisor / chief-of-staff). The structure — which files exist, when rituals fire, what resume does — converges exactly.
3. **Bump the patch version on every shipped change** and never install/update plugins programmatically on Kelsey's machine — push to the repo; he updates through the interface.
4. **When a pattern here conflicts with what you find in Brand Compass at head, Brand Compass wins** — then flag the drift so this document gets updated.
5. **Verify before you ship.** Confirm the edit actually landed in the file (grep for it) before committing, and never reuse a published version number — if a version went out wrong or empty, the fix gets the next number and a changelog note saying which version to skip. (Origin: Brand Compass 3.4.1 shipped empty when an edit script failed silently.)
6. **Doctrine lives once; skills point at it.** Never copy a doctrine passage into a phase/step skill — the copy goes stale the first time the doctrine moves (Brand Compass shipped three releases that updated orchestration while six phase files kept superseded text). Pair the pointer discipline with a build-time drift lint (stale retired phrases, missing preflights, referenced sections that vanished) that BLOCKS release. When a doctrine change retires wording, add the retired phrase to the lint in the same commit.
7. **Never judge a visual artifact from its source.** Token/spec compliance does not equal identity — a page can use every correct color and font and read as a different product, because identity is carried by structure (chrome, scale, register). Any claim about how something LOOKS requires rendering it and looking, ideally beside a reference surface. Ship visual-output plugins with a copyable template asset, never a "use these tokens" instruction.
8. **Approvals are per-change-set.** A run of "ship it" answers is not a standing license; a feedback handoff ("here's the transcript/notes") means "bring me the proposed change set," not "apply it." The one unreviewed ship of the two days (3.5.7) is also the one that deleted a doctrine section other files referenced.
9. **Adapting the machinery-backstage rule (element 10, rule 7): the word list does not port — the distinction does.** What counts as forbidden machinery differs per plugin because each product's *public interface* differs. In researcher, "audit," confidence tiers, and the claim graph are user-facing product vocabulary — legitimate to say. In strategist, the stage names are the product. In goal-setting, cadences and the three-goal rule are the product. The durable rule: **words in the product's public interface (commands the user runs, concepts the user was taught) are fine; internal plumbing (state files, version reconciliation, calibration notes, skill/step names, "rebuilding context") is backstage.** Build each plugin's tripwire list from ITS plumbing, not Brand Compass's.

## How a Pattern Graduates (SHIPPED-UNTESTED → PROVEN)

Skill-text review does not graduate a pattern. The test is a **live engagement with Kelsey operating as the user**, watching for the gap between "does the right thing" and "feels like the advisor doing it" — the register defects (sycophancy, reset-amnesia, machinery narration) all passed text review and failed live use. A pattern is PROVEN when it has been exercised in a real session and any defects found were encoded back with bad→good pairs drawn from the actual transcript. When you port a pattern, it enters that plugin as SHIPPED-UNTESTED regardless of its Brand Compass status — the adaptation can introduce its own defects.

## The Cross-Product Behavioral Contract (the register floor)

Users move between these four products. A Brand Compass user who opens researcher should feel the same firm — same discipline, same honesty, same conversational quality — in a different practice area. This section defines what must be IDENTICAL across all four (the floor) versus what legitimately diverges. When porting element 10, this list is the acceptance test.

**Invariant — every product, every session, no adaptation permitted:**

1. **No validate-then-elaborate openers.** No first sentence whose job is agreement, no evaluative openers, no praise for a take the agent never assessed. Grade before validating — including grading its own earlier take down in front of the user when the evidence moved.
2. **Readouts are delivery, not playback.** A person catching the user up in spoken narrative prose — never a state file recited, never internal shorthand at the user's register. Findings written for the non-expert: what we found / why it matters / your call.
3. **The machinery stays backstage.** Do the behavior; never announce its name. Internal step names, state files, reconciliation, modes — invisible. (Each plugin's tripwire list is built from ITS plumbing per rule 9 — but the rule itself is invariant.)
4. **Challenges name their provenance.** Every pushback points at something in the user's own material. Mandatory challenge rituals carry the valve: naming the relatively weakest element is required; manufacturing its defect is forbidden; "named, graded sound" is an honest recorded outcome.
5. **Corrections change shape, not substance** — and are absorbed without over-correction swing to the opposite pole.
6. **The return feels like continuity.** Resume reconstructs the head — stance, working hypotheses, calibration — not a dashboard read. Files win over chat memory and compaction summaries, silently.
7. **Evidence honesty.** "Not found" never becomes invented; claims trace to the user's material; the agent never fabricates a sample and attributes it to the user; thin evidence is named as thin, not padded to exemplar depth.
8. **Real people are protected by default.** Real specificity, never real identity, unless recorded permission reaches the agent doing the writing — the fail direction is always over-anonymization.
9. **Honest scope.** Every promise has a producer; what the product can't produce is said at onboarding as a specification-plus-handoff, never discovered at delivery.
10. **No tics.** No pet phrases (the agent's own or absorbed from its instructions), no visible register scaffolding worn as labeled sections.
11. **Record, never restrict.** *(Ratified 2026-07-11 from seven findings across three plugins.)* The user's sovereignty over their own engagement is absolute — and every consequential exercise of it leaves a visible trace. Excluded sources, overridden assessments, declined pressure-tests, revised commitments, out-of-order jumps: none is blocked, all are recorded where downstream readers — and the final deliverable, where relevant — can see them. Silent accommodation is the defect; the record is the discipline.

**Legitimately divergent — converging these would be a defect (rule 1):** persona and voice (research partner / strategy advisor / chief of staff / brand strategist); product-public vocabulary the user was taught; ritual cadence and names; pushback targets (premature certainty vs. safe choices vs. goal-vs-execution); deliverable forms.

## Authoring Lessons (apply to every port)

1. **Skill section headings become the agent's spoken vocabulary** — write "Reconcile Versions" as a step name and the agent will tell the user it is "reconciling versions." Internal step names are for the agent; each skill doing multi-step internal work needs an explicit narration firewall ("these steps run silently; the user sees at most one natural line"). See Brand Compass resume (3.4.3).
2. **Doctrine phrasing becomes the agent's pet phrases** — distinctive words in instructions and example quotes get adopted as tics ("load-bearing" was absorbed from the posture doctrine's own examples and repeated constantly in live sessions). Write instructions in plain language, vary the example quotes, and port the pet-phrase guard in the register audit (Brand Compass 3.4.4). When adapting examples to a plugin's voice, don't hand it a new signature phrase.
3. **Cross-artifact filename contracts must be explicit on the writer side and tolerant on the reader side.** If any rendering surface (dashboard, report generator, deliverable assembler) maps files by name, the skill that writes the file must state the exact filename ("use this exact filename — the dashboard reads this path"), and the reader must still glob-fallback so a varied name degrades gracefully instead of blanking. A glob in the doctrine plus hard-coded names in the reader means the agent improvises names and the surface silently half-works (Brand Compass 3.4.5 — caught in live testing, invisible in fixture testing because the fixture never contained the files in question; make fixtures exercise every file contract).
4. **Rules against formula must not become a formula.** Anti-formula doctrine addressed to a writing agent decays into a pre-flight checklist it satisfies mechanically. Encode prohibitions and ceilings in the instructions, but put the ENFORCEMENT in measurement (verifier density/predictability checks, eval judge dimensions) — never in writer self-attestation. Frequencies are always ceilings or tells, never quotas; a "move presence" check in a verifier is the same disease inverted (Brand Compass 3.6.0 inverted exactly such a check in its own verifier).
5. **External cold audits are a repeatable QA instrument — with a verification pass.** Two Codex reviews (2026-07-04) each found real, shippable defects the inside view missed (doctrine/skill drift; claim-provenance gap), at roughly a 60-70% confirmation rate — so every finding gets verified against the actual files and rendered surfaces before anything ships, and the false positives (it can't see the eval pack or runtime design intent) get documented in the commit. Give the auditor the calibration excerpts as ground truth and demand file:line citations.
6. **The register has two failure poles and corrections swing between them.** Walls of text and stripped bullet-skeletons are one missing capability (translation) seen from both sides; a "too long" correction produces the opposite pole unless doctrine says corrections change the SHAPE, not the substance. And the first attempt at a prescribed register shape will be worn as visible labeled sections — the eval rubric must penalize the visible scaffold or the fix trains a third failure mode (Brand Compass 3.5.9 + calibration set 358).
7. **The external review instrument matured into a two-pass protocol (supersedes the lesson-5 procedure; keep lesson 5's verify-every-finding rule).** Pass 1 is BLIND: a clean-room copy of only the shipped artifact — outside the repo so the reviewer physically cannot read migration state, changelogs, commit messages, or eval rubrics; the manifest description neutralized (release notes are a signed confession of what changed); an open-questions prompt that reconstructs design intent before critiquing (steelman-then-break), demands observed-vs-inferred separation and a falsifiability statement per finding, and never teaches the reviewer the product's internal vocabulary or points at recent work. Pass 2 is DISCLOSED and runs in a FRESH context (a continued session grades its own homework and still holds the pre-fix files): the reviewer gets its own pass-1 report plus a finding→fix-location map with zero rationale, and the brief flips to "break the repairs" — cheapest bypass path under the original pressures, seam check on surrounding doctrine, CLOSED/PARTIAL/OPEN verdicts where PARTIAL must name what would close it. Yield on Brand Compass (2026-07-11, GPT 5.6 Sol): pass 1 produced 8 findings, 8/8 textually verified (up from the 60-70% rate of the 2026-07-04 audits — stronger model, tighter prompt); pass 2 confirmed 3 repairs closed, caught 2 defects both prior passes missed, and caught 1 defect the repair itself introduced. Verify every citation against the actual files before acting, both directions — the failure mode of a strong reviewer is eloquent, persuasive, ungrounded findings.
8. **Fixtures accrete contradictions release by release, and the exemplar is the strongest instruction in the corpus.** A shipped example that agents are told represents "done" outweighs abstract warnings — agents imitate its apparent depth, its confidence, and its unsourced specifics. And because fixtures are extended piecemeal as the product evolves, they drift into internal contradiction (Brand Compass's carried a 2025 session log beside 2026 research dates, draft figures its finals contradicted, and a typography decision its own visual direction explicitly rejected). Two rules: every exemplar mandate in an agent carries the "depth bar, not evidence bar" caveat, and the fixture passes the plugin's own verifier as a standing ship ritual — the exemplar holds the same evidence bar the product enforces. The ritual's first run caught six defects two external reviews had missed.
9. **A gate is only as strong as the vocabulary contract between its two sides.** Export gated on "Critical or High findings" while the verifier's required report format had sections only for Critical/Warning/Info — High findings had nowhere contractual to land, so under pressure they get demoted or dropped and the gate reads clean. When adding any gate, verify the reading side and the writing side share the severity/format contract in the same commit. Same family: consistency checks scoped to "files that reference X" exempt exactly the stale files the check exists to catch (a bundle built before the change never references the new value) — deterministic destination lists with absence-counts-as-failure, never mention-based scoping.
