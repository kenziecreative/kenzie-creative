# Redundancy

> Extra capacity or backup paths that keep a system functioning when something fails.

## Description

Redundancy is the deliberate inclusion of backup capacity: spare time, extra cash, alternate suppliers, second reviewers, replicated data, cross-trained staff, fallback plans, or emotional margin.

It can look inefficient when everything goes well. Its value appears when something breaks.

## Mechanism

Redundancy reduces fragility by removing single points of failure. If one component fails, another can absorb the load or buy time for repair.

The tradeoff is cost. Redundancy consumes money, attention, space, speed, and simplicity. The strategic question is where failure would be expensive enough to justify slack.

## When to Use

Use redundancy when failure would cascade, recovery would be slow, or the environment is uncertain enough that the primary path cannot be trusted.

It is especially useful for critical infrastructure, finances, health, operations, project schedules, security, caregiving, and high-trust commitments.

## When It Backfires

Redundancy backfires when it becomes complexity without resilience. Too many backup systems can be hard to maintain, test, or coordinate.

It also backfires when people use redundancy as an excuse not to fix the primary system. Backup capacity should reduce fragility, not hide negligence.

## Practical Use

- Identify single points of failure.
- Ask what failure would cost and how quickly recovery must happen.
- Add slack where the cost of failure exceeds the cost of backup.
- Test whether the backup actually works before it is needed.

## Decision Prompt

Where would one failure cause disproportionate damage, and what backup would prevent collapse without bloating the system?
